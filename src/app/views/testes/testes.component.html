    Detalhes do Projeto

    Introdução

    A Testful é uma empresa de desenvolvimento de software focada em testes para garantir a segurança, confiabilidade e consistência de softwares e implementações.

    Qual a importância de testes?.. Implementação de testes em Softwares garante a integridade , a consistência dos dados e as informações por ele disponibilizadas.

    Objetivos e Propósitos

    ExoAPI (Contratante)

    A ExoApi é uma empresa de desenvolvimento de softwares focada em desenvolver aplicações para empresas de grande e médio porte, além de construção de Web Applications, Web Sites, Local Systems , focando sempre em dar um produto seguro.

    Para isso, o sistema desenvolvido precisa de uma análise de teste em sua aplicação desenvolvida com base numa aplicação de biblioteca, no qual usuários farão busca da disponibilidade de um livro, para isso o usuário necessita ter cadastro de acesso, tendo como segurança de log o uso de token.

    Objetivos

    Com o objetivo da implantação de uma nova plataforma que será utilizada para organização e busca de cadastros de usuários e livros de uma biblioteca, na qual fará cadastro de usuários e livros, bem como a requisição dos mesmos, além da necessidade da alteração desses cadastros, como atualização, disponibilidade, ou remoção dos mesmos.

    Escopo

    A LibApi passará por testes unitários, testes de integração e de sistema. Os testes unitários e de integração vão lidar com a qualidade funcional, das bases de dados, de controle de acesso, enquanto os teste de sistemas tratarão as questões de performance. Os testes de performance entre o banco de dados será :

    • na correta atualização de um cadastro no banco de dados, tanto ao cadastrar como ao atualizar (em todas entidades)
    • criar token ao realizar um login, bem como erro com dados inválidos

    Objetivos do Teste de sistema

    O Teste de sistema tem o objetivo de provar que :

    • A funcionalidade entregue pela equipe de desenvolvimento, está de acordo com as especificações da documentação de requisitos.
    • O software é de alta qualidade; o software vai substituir as/dar suporte às funções de negócio desejadas e alcançar os padrões requisitados pela companhia para o desenvolvimento de novos sistemas.
    • O software entregue faz a interface correta com os sistemas existentes.

    1.3 - Critérios de entrada / saída

    Os critérios de entrada e saída do sistema devem ser atendidos antes que o teste de sistema comece. Por recomendação, solicita-se que haja no mínimo 2 dias a um teste final de integração após a última versão for testada.

    Teste de aceitação :para atingir o critério de aceitação os testes devem ser completados com sucesso, com taxa de 80% de aceitação , isso significa que erros encontrados durante os testes de aceitação não devem impedir que ao menos 80% do teste tenham seus testes completados. Lembrando que esses testes não tem a intenção de fazer um teste profundo do software. Caso havendo a necessidade de suspensão do teste do sistema , critérios de recomeço devem ser especificados, e os teste não devem recomeçar enquanto o software não atender os critérios.

    2 - Tipo teste

    Os testes serão realizados de forma automatizada fazendo uso de uma ferramenta de testes, focando em entregar um software com suas funcionalidades , dando retorno dados com base nas funcionalidades para as quais foram projetadas. É esperado que o benefício total do teste automatizado ocorra somente quando os testes tenham sido executados três ou mais vezes.

    2.4 - Teste de aceitação operacional ( TAO )

    Teste a ser realizado pela equipe de instalação e suporte de sistema, antes de implantar o sistema propriamente dito. Esta equipe vai definir seus próprios critérios de testes e aplicá-los.

    3 - Fases e Ciclos de testes

    3.1 - Ciclos de testes de sistema

    A principal estratégia é testar a aplicação tendo uma liberação caso seu retorno positivos tenha aproximadamente 80% dos erros corrigidos neste período. Com a maioria dos erros consertados, ações padrão usadas serão testadas para provar elementos individuais e o processamento total do sistema na liberação até a fase 3 dos testes.

    Quando os erros ( que potencialmente impactam o processamento geral ) forem consertados, um conjunto adicional de casos de teste são processados na liberação para assegurar que o sistema funcione de maneira integrada. Não deve haver erros antes do início da fase 4 de testes para a liberação.

    3.2 - Entrega de software

    Durante o teste de sistema, a liberação de novas versões do software deve ser coordenadas entre o líder da equipe de desenvolvimento e o analista de negócio de teste de sistema. No entanto caso haja erros de alta precisão, novas versões só devem ser liberadas quando os objetivos concordados sejam alcançados ( a versão seguinte não deve conter erros, mas deve-se realizar a correção de erros e bugs antes da liberação do mesmo).

    4 - Gerenciamento de erro & gerenciamento de configuração

    Erros válidos - esses serão concordados e categorizados pela equipe de revisão de erros da seguinte forma:

    • A) ERROS SÉRIOS - que impedem a continuidade do teste de sistema de função em particular, graves erros de dados
    • B) ERROS RELATIVOS - dados sérios que não impedem a implementação
    • C) PEQUENOS ERROS - erros que não impedem nem afetam a funcionalidade

    Caso os erros seja sérios devem ser eliminados pela equipe de correções de bugs em 48hrs a partir da detecção do erro. Se erro impeça o teste de sistema de continuar, a eliminação deve ser dentro de 4 hrs.

    Caso os erros seja relativos, deve ser solucionado em 24 hrs e os pequenos erros podem ser solucionados em até 3 dias. No entanto a liberação de novas versões do software devem ser coordenadas com o analista de teste - novas versões devem ser liberadas quando concordadas, que haja um benefício definitivo com correções e sem bugs.

    Classificação de Bugs:

    • A) Que travam o sistema:

      • Dados incorretos são passados ao sistema em corrompimento ou travamento do sistema
      • Retorno incorreto de dados

    • B) classificados relevantes quando:

      • Afetam um elemento menos importante da funcionalidade. Ex: um default de um valor não está correto e precisa ser corrigido manualmente.
      • dados afetados não tem um grande impacto. Ex: um dado enviado de forma incorreta

      • C) Classificados como inofensivos:

      • ex: opções em duplicidade

    Procedimento para manutenção de sistema de registro de erros

    • Evita esforço desnecessário dos testadores
    • Põe o desenvolvedor imediatamente a par do problema
    • Permite que o desenvolvedor adicione mecanismos necessários para localizar o erro
